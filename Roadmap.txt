SYSTEM VISION, ARCHITECTURE, AND GLOBAL DEVELOPMENT RULES

------------------------------------------------------------
1. PLATFORM OVERVIEW
------------------------------------------------------------

Platform Name: SERVICHAYA (सर्विचाया)

Meaning:
Service + Chhaya (Protection/Care) - A trusted ecosystem of services and workforce

Platform Goal:
To build a trusted ecosystem where customers can:

1) Book instant services (repair, maintenance etc.)
2) Hire workforce (maid, gardener, driver, etc.)
3) Manage home service history
4) Interact with verified service professionals
5) Track service lifecycle

The platform will support three primary user groups:

CUSTOMER
SERVICE PROVIDER
PLATFORM ADMIN

Future users may include:

SUPPLIERS
WORKFORCE CONTRACTORS
PARTNERS

The platform must remain scalable and generic so that
new services, workforce types, and cities can be added
without code changes.

------------------------------------------------------------
2. DOMAIN STRUCTURE
------------------------------------------------------------

All applications share the SAME CODE BASE.

Different domains activate different features using RBAC.

Domains:

customer.servixa.com
partner.servixa.com
admin.servixa.com

Each domain loads the same backend but UI components and features are controlled by ROLE.

------------------------------------------------------------
3. CUSTOMER ACCESS MODEL
------------------------------------------------------------

Customer application must remain PUBLIC.

Users should be able to explore the platform without registration.

Customer registration must only be required when the user performs
a critical action such as:

Posting a job
Saving address
Booking a service
Hiring workforce

This removes friction and improves onboarding.

Customer exploration features available without login:

Browse services
View technicians
View workforce profiles
View pricing ranges
View city coverage

Registration flow must appear only when required.

------------------------------------------------------------

Each module must contain two packages.

Request DTO

Interface Controllers

SERVICE MODULE CONTAINS:

Controller Implementation

Repository

------------------------------------------------------------
7. INTER-MODULE COMMUNICATION RULE
------------------------------------------------------------

Modules must depend ONLY on API modules.

Example:

booking-service
depends on
user-api

Never depend on:

user-service

Feign client defined in API module.

Controller in service module implements same interface.

Example:

UserController (API interface)

@RestController
UserControllerImpl implements UserController

Feign client calls UserController interface.

------------------------------------------------------------
8. FEIGN + LOCAL CONTROLLER BEHAVIOR
------------------------------------------------------------

If only API dependency added:

Feign client used.

If SERVICE dependency added:

Local controller implementation used.

This allows easy migration to microservices.

12. POD DATA ACCESS RULE
------------------------------------------------------------

Users should only see relevant data.

Example:

If Service Provider works in Pod A
they must not see Pod B data.

If Admin manages City A
they should not see City B data.

Benefits:

Reduced database load
Improved performance
Better data security
Simplified UI

------------------------------------------------------------
14. PERFORMANCE RULES
------------------------------------------------------------

System must follow strict performance practices.

RULE 1

Load only required fields using Interface Projections.

RULE 2

Use Native Queries for complex joins.

RULE 3

Use database views when query complexity increases.

RULE 4

Avoid N+1 queries.

RULE 5

Use pagination everywhere.

RULE 6

Never load full tables.

RULE 7

Filtering must be done in backend.

UI must never filter datasets locally.

------------------------------------------------------------
15. SOLID PRINCIPLES
------------------------------------------------------------

System must follow SOLID principles.

Single Responsibility
Open Closed
Liskov Substitution
Interface Segregation
Dependency Inversion

Business logic must stay inside SERVICE layer only.

Controllers must remain thin.

Configuration Management:

Admin panel must provide UI for:
- Editing UI configuration (color pickers, sliders, preview)
- Managing business rules (forms with validation)
- Toggling feature flags (switches, rollout sliders)
- Testing configuration changes before applying

Backend Implementation:

All configuration must be cached (Redis) for performance.
Cache invalidation on configuration updates.

Configuration loading:
- On application startup
- On configuration change (webhook/event)
- Lazy loading with fallback to defaults

Frontend Implementation:

UI configuration loaded on app initialization.
Applied via CSS variables or theme provider.
Hot-reload on configuration changes (if possible).

Example:

Backend: ConfigurationService.getUIConfig('primary_color')
Frontend: Apply to CSS variable --primary-color
Result: Theme changes without code deployment

DEFAULT MATCHING RULES

Rule 1: Skill Match (Weight: 30%)

Perfect skill match: 100 points
Partial skill match: 60 points
No skill match: 0 points (excluded)

Rule 2: Distance (Weight: 25%)

Within POD: 100 points
Within Zone: 70 points
Within City: 40 points
Outside City: 0 points (excluded)

Rule 3: Provider Rating (Weight: 20%)

5 stars: 100 points
4.5 stars: 90 points
4 stars: 80 points
3.5 stars: 60 points
Below 3.5: 0 points (excluded)

Rule 4: Subscription Tier (Weight: 15%)

Enterprise: 100 points
Professional: 80 points
Basic: 60 points
Free: 40 points

Rule 5: Acceptance Rate (Weight: 10%)

Above 90%: 100 points
80-90%: 80 points
70-80%: 60 points
Below 70%: 40 points

--------------------------------------------------------------------

MATCHING SCORE CALCULATION

Final Score = Σ (Rule Score × Rule Weight)

Example

Provider A:
Skill Match: 100 × 30% = 30
Distance: 100 × 25% = 25
Rating: 90 × 20% = 18
Subscription: 80 × 15% = 12
Acceptance: 100 × 10% = 10
Total Score: 95

Provider B:
Skill Match: 100 × 30% = 30
Distance: 70 × 25% = 17.5
Rating: 100 × 20% = 20
Subscription: 100 × 15% = 15
Acceptance: 80 × 10% = 8
Total Score: 90.5

Provider A ranks higher.

MATCHING FILTERS (MUST PASS)

Providers must pass all filters to be considered:

1) Active status
2) Available (not blocked)
3) Within serviceable POD/Zone
4) Has required skill
5) Rating >= 3.0

CANCELLATION RULES AND FEES

Rule 1: Customer Cancellation Before Provider Accepts

Fee: ₹0 (Free cancellation)
Refund: 100% of payment (if paid)
Action: Job status → CANCELLED
Provider: No penalty

Rule 2: Customer Cancellation After Provider Accepts (Before Start)

Fee: 10% of job amount (minimum ₹50)
Refund: 90% of payment
Action: Job status → CANCELLED
Provider: Receives 50% of cancellation fee as compensation

Rule 3: Customer Cancellation After Provider Started

Fee: 20% of job amount (minimum ₹100)
Refund: 80% of payment
Action: Job status → CANCELLED
Provider: Receives 50% of cancellation fee + 50% of work done value

Rule 4: Provider Cancellation After Acceptance

Fee: 5% penalty on provider earnings
Refund: 100% to customer
Action: Job status → CANCELLED
Provider: Rating penalty, possible suspension if repeated

Rule 5: Provider No-Show (Doesn't arrive within 30 minutes)

Fee: 10% penalty on provider
Refund: 100% to customer
Action: Job status → CANCELLED
Provider: Severe rating penalty, possible suspension

Rule 6: Customer No-Show (Not available when provider arrives)

Fee: ₹0 (No fee)
Refund: 100% to customer
Action: Job status → CANCELLED
Provider: Receives travel compensation (₹100-200 based on distance)
